Mórchlár uses PostgreSQL as its primary database with Drizzle ORM for type-safe database operations. This document provides a comprehensive overview of all database tables, columns, relationships, and constraints.

## Schema overview

The database schema is organized into five main domains:

1. **Authentication** - User accounts, sessions, and OAuth
2. **Groups** - Team/organization management
3. **Group members** - Group membership with role-based access
4. **Projects** - GitHub repository-linked projects
5. **Tasks** - Hierarchical task management

## Authentication tables

These tables are generated by better-auth and handle user authentication, sessions, and OAuth accounts.

### user

Stores user account information.

**Source**: `lib/db/schema/auth.ts:8-23`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `text` | PRIMARY KEY | Unique user identifier |
| `name` | `text` | NOT NULL | User's display name |
| `email` | `text` | NOT NULL, UNIQUE | User's email address |
| `emailVerified` | `boolean` | NOT NULL, DEFAULT false | Email verification status |
| `image` | `text` | nullable | User's profile image URL |
| `createdAt` | `timestamp` | NOT NULL, DEFAULT now() | Account creation timestamp |
| `updatedAt` | `timestamp` | NOT NULL, DEFAULT now() | Last update timestamp (auto-updates) |
| `selectedGroup` | `integer` | nullable, FK → groups.id | Currently selected group for UI state |
| `selectedProject` | `integer` | nullable, FK → projects.id | Currently selected project for UI state |

**Foreign key constraints**:
- `selectedGroup` → `groups.id` (ON DELETE SET NULL)
- `selectedProject` → `projects.id` (ON DELETE SET NULL)

**Relations**:
- Has many `session` records
- Has many `account` records (OAuth providers)
- Belongs to many `groups` through `groupMembers`

### session

Stores active user sessions.

**Source**: `lib/db/schema/auth.ts:25-42`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `text` | PRIMARY KEY | Session identifier |
| `expiresAt` | `timestamp` | NOT NULL | Session expiration time |
| `token` | `text` | NOT NULL, UNIQUE | Session token (stored in HTTP-only cookie) |
| `createdAt` | `timestamp` | NOT NULL, DEFAULT now() | Session creation time |
| `updatedAt` | `timestamp` | NOT NULL | Last session update (auto-updates) |
| `ipAddress` | `text` | nullable | IP address of session creation |
| `userAgent` | `text` | nullable | User agent string |
| `userId` | `text` | NOT NULL, FK → user.id | User who owns this session |

**Indexes**:
- `session_userId_idx` on `userId`

**Foreign key constraints**:
- `userId` → `user.id` (ON DELETE CASCADE)

**Relations**:
- Belongs to one `user`

### account

Stores OAuth provider accounts linked to users.

**Source**: `lib/db/schema/auth.ts:44-66`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `text` | PRIMARY KEY | Account identifier |
| `accountId` | `text` | NOT NULL | Provider-specific account ID |
| `providerId` | `text` | NOT NULL | OAuth provider identifier (e.g., 'github') |
| `userId` | `text` | NOT NULL, FK → user.id | User who owns this account |
| `accessToken` | `text` | nullable | OAuth access token |
| `refreshToken` | `text` | nullable | OAuth refresh token |
| `idToken` | `text` | nullable | OpenID Connect ID token |
| `accessTokenExpiresAt` | `timestamp` | nullable | Access token expiration |
| `refreshTokenExpiresAt` | `timestamp` | nullable | Refresh token expiration |
| `scope` | `text` | nullable | OAuth scopes granted |
| `password` | `text` | nullable | Hashed password (for non-OAuth auth) |
| `createdAt` | `timestamp` | NOT NULL, DEFAULT now() | Account creation time |
| `updatedAt` | `timestamp` | NOT NULL | Last update time (auto-updates) |

**Indexes**:
- `account_userId_idx` on `userId`

**Foreign key constraints**:
- `userId` → `user.id` (ON DELETE CASCADE)

**Relations**:
- Belongs to one `user`

<Note>
The `accessToken` is used to make GitHub API calls on behalf of the user with the scopes `user:email` and `repo`.
</Note>

## Groups tables

### groups

Stores team/organization information.

**Source**: `lib/db/schema/groups.ts:7-15`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `serial` | PRIMARY KEY | Auto-incrementing group identifier |
| `name` | `text` | NOT NULL | Group display name |
| `slug` | `text` | NOT NULL, UNIQUE | URL-friendly slug (auto-generated) |
| `createdAt` | `timestamp` | NOT NULL, DEFAULT now() | Group creation time |
| `updatedAt` | `timestamp` | NOT NULL | Last update time (auto-updates) |

**Relations**:
- Has many `groupMembers` (members of this group)
- Has many `projects`

**Validation schema** (from `lib/db/schema/groups.ts:21-26`):
```typescript
export const InsertGroup = createInsertSchema(groups).omit({
    id: true,
    slug: true,
    createdAt: true,
    updatedAt: true,
});
```

The `slug` is automatically generated from the `name` when creating a group.

### groupMembers

Junction table for group membership with role-based access control.

**Source**: `lib/db/schema/groupMembers.ts:10-16`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `groupId` | `integer` | NOT NULL, FK → groups.id, PK | Group identifier |
| `userId` | `text` | NOT NULL, FK → user.id, PK | User identifier |
| `role` | `groupRole` | NOT NULL | Member's role in the group |

**Primary key**: Composite key on `(groupId, userId)`

**Foreign key constraints**:
- `groupId` → `groups.id` (ON DELETE CASCADE)
- `userId` → `user.id` (ON DELETE CASCADE)

**Relations**:
- Belongs to one `group`
- Belongs to one `user`

### groupRole enum

**Source**: `lib/db/schema/groupMembers.ts:8`

The `groupRole` PostgreSQL enum defines three permission levels:

- **`owner`** - Full access including group deletion
- **`developer`** - Can create/edit projects and tasks
- **`reader`** - Read-only access

```typescript
export const groupRole = pgEnum('groupRole', ['owner', 'developer', 'reader']);
```

## Projects table

### projects

Stores project information linked to GitHub repositories.

**Source**: `lib/db/schema/projects.ts:7-18`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `serial` | PRIMARY KEY | Auto-incrementing project identifier |
| `title` | `text` | NOT NULL | Project display name |
| `groupId` | `integer` | NOT NULL, FK → groups.id | Group that owns this project |
| `repoId` | `integer` | NOT NULL | GitHub repository numeric ID |
| `repoName` | `text` | NOT NULL | GitHub repository name |
| `repoOwner` | `text` | NOT NULL | GitHub repository owner |
| `createdAt` | `timestamp` | NOT NULL, DEFAULT now() | Project creation time |
| `updatedAt` | `timestamp` | NOT NULL | Last update time (auto-updates) |

**Foreign key constraints**:
- `groupId` → `groups.id` (implicit ON DELETE CASCADE)

**Relations**:
- Belongs to one `group`
- Has many `tasks`

**Validation schemas**:

Two insert schemas are provided for different contexts:

1. **Server-side** (`lib/db/schema/projects.ts:32-36`):
   ```typescript
   export const InsertProject = createInsertSchema(projects).omit({
       id: true,
       createdAt: true,
       updatedAt: true,
   });
   ```

2. **Client-side** (`lib/db/schema/projects.ts:41-48`) - Accepts repo URL instead of numeric ID:
   ```typescript
   export const ClientInsertProject = createInsertSchema(projects).omit({
       id: true,
       createdAt: true,
       updatedAt: true,
       repoId: true,
   }).extend({
       repo: z.string(),
   });
   ```

The server resolves the `repo` string to `repoId`, `repoName`, and `repoOwner` using the GitHub API.

## Tasks table

### tasks

Stores hierarchical task/milestone information with timeline data.

**Source**: `lib/db/schema/tasks.ts:7-25`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `serial` | PRIMARY KEY | Auto-incrementing task identifier |
| `title` | `text` | NOT NULL | Task title |
| `description` | `text` | nullable | Detailed task description |
| `projectId` | `integer` | NOT NULL, FK → projects.id | Project this task belongs to |
| `parentId` | `integer` | nullable, FK → tasks.id | Parent task (for subtasks) |
| `startTime` | `timestamp` | NOT NULL | Task start date/time |
| `endTime` | `timestamp` | NOT NULL | Task end date/time |
| `progress` | `real` | DEFAULT 0 | Completion progress (0.00 to 1.00) |
| `order` | `integer` | DEFAULT 0 | Display order among siblings |
| `createdAt` | `timestamp` | NOT NULL, DEFAULT now() | Task creation time |
| `updatedAt` | `timestamp` | NOT NULL | Last update time (auto-updates) |

**Foreign key constraints**:
- `projectId` → `projects.id` (ON DELETE CASCADE)
- `parentId` → `tasks.id` (ON DELETE CASCADE) - Self-referential for subtasks

**Relations**:
- Belongs to one `project`
- Belongs to one `parent` task (optional)
- Has many `subtasks` (self-referential)

**Hierarchical structure**:

Tasks support unlimited nesting via the `parentId` column. When `parentId` is `NULL`, the task is a top-level task. Otherwise, it's a subtask of the referenced parent.

From `lib/db/schema/tasks.ts:27-40`:
```typescript
export const taskRelations = relations(tasks, ({ one, many }) => ({
    project: one(projects, {
        fields: [tasks.projectId],
        references: [projects.id],
    }),
    parent: one(tasks, {
        fields: [tasks.parentId],
        references: [tasks.id],
        relationName: 'subtasks',
    }),
    subtasks: many(tasks, {
        relationName: 'subtasks',
    }),
}));
```

**Validation schema** (`lib/db/schema/tasks.ts:51-58`):

```typescript
export const InsertTask = createInsertSchema(tasks, {
    startTime: () => preprocessDate,
    endTime: () => preprocessDate,
}).omit({
    id: true,
    createdAt: true,
    updatedAt: true,
});
```

The schema includes custom date preprocessing to convert string dates from the client into Date objects.

## Database migrations

Mórchlár uses Drizzle Kit for schema migrations.

### Running migrations

Generate migration files from schema changes:
```bash
bun run db:generate
```

Apply pending migrations:
```bash
bun run db:migrate
```

### Migration files

Migrations are stored in `lib/db/migrations/` with:
- SQL migration files (e.g., `0001_migration.sql`)
- Metadata snapshots in `meta/`
- Migration journal in `meta/_journal.json`

## Type safety

All database operations are fully type-safe through Drizzle ORM:

### Inferred types

Each schema exports TypeScript types:

```typescript
// From schema definitions
export type UserSchema = typeof user.$inferSelect;
export type GroupSchema = typeof groups.$inferSelect;
export type ProjectSchema = typeof projects.$inferSelect;
export type TasksSchema = typeof tasks.$inferSelect;

// From insert schemas
export type InsertGroupSchema = z.infer<typeof InsertGroup>;
export type InsertProjectSchema = z.infer<typeof InsertProject>;
export type InsertTaskSchema = z.infer<typeof InsertTask>;
```

### Usage in queries

Example query with full type inference (from `lib/db/queries/`):

```typescript
import db from '../index';
import { tasks, projects } from '../schema';
import { eq } from 'drizzle-orm';

// Return type is automatically inferred
export async function getTasksByProject(projectId: number) {
    return await db
        .select()
        .from(tasks)
        .where(eq(tasks.projectId, projectId));
}
```

## Best practices

### Cascading deletes

The schema uses cascading deletes to maintain referential integrity:

- Deleting a user cascades to their sessions and accounts
- Deleting a group cascades to group members
- Deleting a project cascades to all its tasks
- Deleting a task cascades to all its subtasks

### Timestamps

All tables include:
- `createdAt` - Set once on creation with `defaultNow()`
- `updatedAt` - Auto-updates on every change via `$onUpdate(() => new Date())`

### Snake case conversion

The database uses snake_case for column names, automatically converted to/from camelCase in TypeScript:

```typescript
// TypeScript code
const user = { selectedGroup: 1 };

// Database column
// selected_group
```

Configured in `lib/db/index.ts:10` and `drizzle.config.ts:8`.

### Validation

All insert operations are validated using Zod schemas derived from Drizzle schemas:

```typescript
import { InsertTask } from '../schema';

// Automatically validates types, required fields, etc.
const result = InsertTask.parse({
    title: 'New task',
    projectId: 1,
    startTime: new Date(),
    endTime: new Date(),
});
```
